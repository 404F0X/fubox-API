# Fubox API Code-first Design TODO

更新日期：2026-06-12  
状态：当前唯一执行入口  
原则：产品功能和界面设计实现优先；本地可运行、代码可合并、用户路径可点击优先；生产可行性、发布证据和长审查只做后置 backlog。

## 为什么重写 TODO

项目已经有较多 Agent coordination、paid beta、release evidence、production gate 文档。它们对发布审计有价值，但在当前阶段会反复把 Agent 拉回“审查/验证/证明”循环，拖慢真正的产品功能实现。

当前目标是把 fubox API 做成一个可运行、可演示、可继续迭代的 New API 风格 AI Gateway MVP：

```text
Admin 配置供应商/通道/模型/路由
User 注册/登录/兑换额度/创建 API Key
Client 调用 OpenAI-compatible Gateway
Admin/User 查看余额、用量、请求日志、trace 和错误原因
```

本 TODO 覆盖设计重做、页面信息架构、API/DTO 支撑、网关兼容层、计费额度和导入工具的功能实现。旧 TODO 全部归档在 `TODO/archive/2026-06-12_legacy_review_and_production_gate_todos/`。

## Agent 执行规则

### 必须做

- 优先提交能被用户看到、点到、调用到的功能。
- 使用本地 mock provider、本地 Docker/Compose、fixture 或 dry-run 继续推进，不等待真实生产凭证。
- 后端接口先给稳定 DTO、错误码和 secret-safe 响应；前端再围绕这些 DTO 做工作台体验。
- UI 修改以“后台工作台”为目标：表格、筛选、状态 chip、drawer、批量动作、空状态、明确下一步。
- 每个切片只要求最小可信检查：类型检查、相关单测、focused smoke、手工路径说明或截图证据任选其必要组合。
- 发现外部依赖时，先加配置 seam、mockable adapter、占位状态和清晰错误文案，然后继续完成本地路径。
- 安全红线仍然保留：不回显 provider key、Authorization、voucher raw code、prompt/raw payload、secret/idempotency key。
- 2026-06-12：`enterprise-oidc-jwks-validator-executor-v1` 已补：新增 `POST /admin/enterprise/identity-connections/oidc/jwks-validator-executor`，只接受 fixture-safe JWT header/claims summary、kid/alg/aud/iss/exp/nonce presence/hash、JWKS key fingerprint presence 和 `signature_verified` boolean；返回 `validator_result`、`blocked_reasons`、binding/session handoff readiness 和 omitted fields。不接真实 IdP/JWKS 网络、不创建真实外部 session、不回显 raw token/raw claims/JWKS/JWK body/client_secret/Authorization/private key/cert。该项标 `[!]`：逻辑完成但真实 IdP/JWKS 网络 validator、真实 provider key validation 和 live evidence 仍后置。

### 状态标记

- `[x]`：代码/逻辑已完成，当前本地或 mock 路径可消费。
- `[!]`：代码/逻辑已完成，但受真实商户、外部凭证、生产环境、发布审计或人工策略阻塞，当前不再作为功能进度 blocker。
- `[~]`：仍需要继续写代码或补主要业务逻辑；优先派 Agent 清掉。
- `[ ]`：未开始或方向尚未落地。

### 不要做

- 不要把 production readiness、paid beta、release evidence、manifest freshness、clean clone、secret-scan 全量报告作为当前任务 blocker。
- 不要为了“证明可上线”新增大脚本、大 artifact 或新的审查清单。
- 不要因为缺真实支付商户、订阅 scheduler、OIDC/SAML、生产 ClickHouse/tokenizer、外部 provider key 就停止功能开发。
- 不要继续扩写旧的 Agent 状态同步文档；需要记录时只更新本 TODO 的完成状态。
- 不要为了保持旧测试文本而牺牲新的信息架构。

### 当前验收口径

一个功能切片可以合格，只要满足：

1. 用户路径能在本地或 mock 环境跑通，或页面能真实渲染并调用已有 API。
2. 涉及 secret 的响应和 UI 均为脱敏/一次性展示/默认不展示 raw payload。
3. touched area 的类型检查或相关测试通过；如果环境失败，只记录环境问题，不升级成产品 blocker。
4. README 或页面文案能告诉下一个开发者如何点开/调用该功能。

生产级验证放入 P3，不作为 P0/P1 功能实现的停止条件。
