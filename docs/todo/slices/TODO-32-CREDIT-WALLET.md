## Shared Requirements

- Money values MUST be decimal strings with explicit currency, for example `"10.00000000"` and `"USD"`. Do not use float values.
- Every write endpoint MUST require an idempotency key. The same key MUST replay the original result or reject conflicts.
- Every write endpoint MUST emit audit metadata with `actor_id`, `actor_type`, `reason`, `request_id` or `operation_id`, and secret-safe output.
- API responses MUST omit raw payment provider secrets, bearer tokens, database URLs, raw environment values, and raw idempotency material.
- Wallet snapshots are read models only. Credit creation, expiry, revocation, imports, and adjustments MUST write a ledger/admin adjustment/opening entry or credit grant record; they MUST NOT directly mutate wallet snapshot balance as the accounting source of truth.
- Responses SHOULD include `ledger_entry_id`, `credit_grant_id`, `wallet_id`, `tenant_id`, `project_id`, `currency`, `created_at`, `updated_at`, and `staleness`/`readback` markers where applicable.

## POST /billing/credit-grants

Purpose: create a bounded credit grant for a tenant/project/wallet.

Input key fields:

- `idempotency_key`: required string.
- `tenant_id`: required UUID.
- `project_id`: optional UUID.
- `wallet_id`: required UUID.
- `currency`: required ISO currency string.
- `amount`: required decimal string.
- `grant_type`: enum such as `promo`, `manual_credit`, `top_up`, `migration`, `support`.
- `valid_from`, `expires_at`: timestamp window.
- `reason`: required operator/user-facing reason.
- `metadata`: bounded JSON object; no secrets.

Output key fields:

- `credit_grant_id`.
- `status`: `active`, `scheduled`, or `refused`.
- `remaining_amount`.
- `ledger_entry_id` or `opening_entry_id` when the grant is represented by ledger/admin adjustment.
- `audit_id`.
- `idempotency`: `applied`, `replayed`, or `conflict_refused`.
- `secret_safe`: true.

Rules:

- A grant creation that changes spendable credit MUST be backed by a credit grant row and, where it changes accounting balance, a ledger/admin adjustment or opening entry.
- Duplicate `idempotency_key` with identical body replays; different body rejects with `idempotency_conflict`.
- Expired or zero/negative grants are refused.

## GET /billing/credit-grants

Purpose: list credit grants and remaining grant balances for a tenant/project/wallet.

Input key fields:

- `tenant_id`: required UUID.
- `project_id`: optional UUID.
- `wallet_id`: optional UUID.
- `status`: optional filter: `active`, `expired`, `revoked`, `scheduled`.
- `currency`: optional filter.
- `limit`, `cursor`: pagination.

Output key fields:

- `items[]`: `credit_grant_id`, `wallet_id`, `currency`, `original_amount`, `remaining_amount`, `status`, `valid_from`, `expires_at`, `created_at`, `updated_at`, `source`, `last_ledger_entry_id`.
- `next_cursor`.
- `readback`: `fresh`, `stale`, or `estimated`.
- `secret_safe`: true.

Rules:

- The list API must not expose redeem codes, payment provider payloads, raw metadata secrets, or raw idempotency keys.
- Remaining amounts must be derived from grant/ledger readback, not dashboard aggregate alone.

## POST /billing/credit-grants/{credit_grant_id}/expire

Purpose: expire a credit grant at or before its configured expiry.

Input key fields:

- `idempotency_key`: required string.
- `credit_grant_id`: path UUID.
- `effective_at`: optional timestamp.
- `reason`: required string.
- `actor_id`, `actor_type`: required audit fields.

Output key fields:

- `credit_grant_id`.
- `status`: `expired`.
- `remaining_amount_before`.
- `remaining_amount_after`.
- `ledger_entry_id` or `adjustment_entry_id` if expiring the grant requires accounting reversal.
- `audit_id`.
- `idempotency`.
- `secret_safe`: true.

Rules:

- Expiry MUST be idempotent.
- If expiry removes spendable balance, it MUST write a ledger/admin adjustment or explicit grant-consumption record. It MUST NOT directly edit wallet snapshot balance.

## POST /billing/credit-grants/{credit_grant_id}/revoke

Purpose: revoke a credit grant due to fraud, operator correction, refund, chargeback, or support action.

Input key fields:

- `idempotency_key`: required string.
- `credit_grant_id`: path UUID.
- `reason`: required enum/string.
- `revoke_remaining_only`: boolean, default true.
- `actor_id`, `actor_type`: required audit fields.

Output key fields:

- `credit_grant_id`.
- `status`: `revoked`.
- `revoked_amount`.
- `remaining_amount_before`.
- `remaining_amount_after`.
- `ledger_entry_id` or `adjustment_entry_id`.
- `audit_id`.
- `idempotency`.
- `secret_safe`: true.

Rules:

- Revocation that changes spendable balance MUST be backed by ledger/admin adjustment or grant-consumption accounting.
- Revoke must reject attempts that would create negative remaining grant balance unless explicitly represented as a separate debit adjustment with audit.

## GET /billing/wallets/{wallet_id}/remaining-balance

Purpose: user-facing balance summary with enough readback detail to explain spendable credit.

Input key fields:

- `wallet_id`: path UUID.
- `tenant_id`: required UUID.
- `project_id`: optional UUID.
- `currency`: required string.
- `include_grants`: optional boolean.
- `include_budgets`: optional boolean.
- `include_pending`: optional boolean.

Output key fields:

- `wallet_id`, `tenant_id`, `project_id`, `currency`.
- `wallet_available_balance`: decimal string.
- `active_credit_grant_total`: decimal string.
- `pending_reserve_total`: decimal string.
- `confirmed_ledger_effect`: decimal string.
- `budget_remaining`: optional per tenant/project/virtual_key dimension.
- `available_to_spend`: decimal string.
- `last_ledger_entry_id`, `last_credit_grant_id`.
- `readback`: `strongly_consistent`, `fresh`, `stale`, or `estimated`.
- `calculation`: stable formula string.
- `secret_safe`: true.

Rules:

- The API must disclose whether the response is strongly consistent or a read model snapshot.
- The API must not imply strong balance consistency if the read path is dashboard aggregate or stale cache.
- It must expose enough ids to reconcile with ledger entries without exposing secrets.

## POST /billing/opening-balance-imports

Purpose: import New API or external existing balance into the ledger/wallet system.

Input key fields:

- `idempotency_key`: required string.
- `tenant_id`: required UUID.
- `project_id`: optional UUID.
- `wallet_id`: required UUID.
- `currency`: required string.
- `opening_amount`: required decimal string.
- `external_source`: enum/string, for example `new_api_balance`.
- `external_reference_id`: required string.
- `effective_at`: timestamp.
- `reason`: required string.
- `actor_id`, `actor_type`: required audit fields.

Output key fields:

- `opening_import_id`.
- `wallet_id`.
- `ledger_entry_id` or `admin_adjustment_entry_id`.
- `amount`, `currency`.
- `status`: `imported`, `replayed`, or `refused`.
- `audit_id`.
- `idempotency`.
- `secret_safe`: true.

Rules:

- Opening balance import MUST write an opening ledger entry or admin adjustment entry. It MUST NOT directly mutate wallet snapshot.
- The import must be idempotent by `idempotency_key` and should also refuse duplicate `external_source + external_reference_id` unless replaying the same import.

## POST /billing/admin-adjustments

Purpose: explicit operator adjustment path for credits/debits that are not customer self-serve payment events.

Input key fields:

- `idempotency_key`: required string.
- `adjustment_operation_id`: required UUID.
- `tenant_id`: required UUID.
- `project_id`: optional UUID.
- `wallet_id`: required UUID.
- `related_ledger_entry_id`: optional UUID.
- `amount`: required signed decimal string.
- `currency`: required string.
- `adjustment_kind`: `credit` or `debit`.
- `reason`: required string.
- `actor_id`, `actor_type`: required audit fields.

Output key fields:

- `ledger_entry_id`.
- `adjustment_operation_id`.
- `status`: `applied`, `replayed`, or `refused`.
- `amount`, `currency`.
- `wallet_check`, `budget_checks` for debit adjustments.
- `audit_id`.
- `idempotency`.
- `secret_safe`: true.

Rules:

- Admin adjustments MUST use the ledger/admin adjustment path.
- Debit adjustments MUST pass wallet/budget checks or refuse conservatively.
- Credit adjustments MUST be visible in remaining balance readback and reconciliation.

## Contract Test Targets

- Create grant idempotent replay and conflict refusal.
- List grants omits raw idempotency keys and secrets.
- Expire/revoke writes accounting effect when spendable balance changes.
- Remaining balance summary matches wallet + active grants + pending/confirmed ledger + budgets.
- Opening balance import writes opening ledger/admin adjustment entry and rejects direct wallet mutation.
- Admin adjustment credit/debit maps to existing consistent writer primitives and remains secret-safe.
